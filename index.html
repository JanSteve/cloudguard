AWS Shared Responsibility Model

The foundational principle of cloud security.

🟢 AWS Manages This
Security OF the Cloud
🖥️
Physical Hardware & Data Centers
AWS secures the physical buildings, servers, and networking equipment across its global regions.
🌐
Network Infrastructure
The underlying network between data centers, availability zones, and regions is entirely AWS-managed.
🔌
Hypervisor & Virtualization
The software that creates and manages virtual machines runs on AWS hardware and is patched by AWS.
🏢
AWS Global Infrastructure
Regions, Availability Zones, and Edge Locations are designed and maintained for reliability and security.
🛡️
Managed Service Internals
Services like S3 and RDS — AWS handles the operating system and platform patches internally.
YOU & AWS
🔵 You Manage This
Security IN the Cloud
🔑
IAM Users, Roles & Permissions
You decide who can access your AWS account and what they can do.
🗄️
Data Encryption
Encrypting your data at rest and in transit is your responsibility — AWS provides the tools (KMS).
🌐
VPC & Security Groups
Configuring your virtual private cloud, firewalls, and network access controls is on you.
💻
OS Patching (EC2)
When you run EC2, you're responsible for keeping the operating system updated and secure.
🔧
Application Security
Your application code, dependencies, and configurations must be secured by you.
📋
Data Classification & Compliance
Classifying your data, meeting regulatory requirements, and ensuring compliance is your job.
💡

The Rule of Thumb

AWS secures the infrastructure. You secure everything you build ON it. If you can configure it in the AWS console, it's probably your responsibility.

📋

Exam Focus

This is one of the most tested concepts on the AWS Cloud Practitioner exam. Understand the boundary — hardware is always AWS, your data is always you.

🏦

Enterprise Reality

In the financial sector, auditors ask about this model constantly. Understanding it shows security maturity beyond most candidates.

Identity
AWS IAM
AWS's gatekeeper for who can do what.
Controls who can access what in your AWS account. Every API call, every login, every permission — IAM decides if it's allowed. Without IAM, there is no cloud security.
💡 Think of IAM like a hotel key card system — different cards open different doors, and the front desk (AWS) decides who gets what.
  • Controlling developer access to production
  • Granting EC2 instances access to S3
  • Enforcing multi-factor authentication

💡 The bouncer at a club who checks IDs for your application — only lets in verified users.

Your 24/7 threat radar.
Continuously monitors your AWS account for suspicious activity and threats, 24/7, without you doing anything. Uses machine learning to detect unusual API calls, unauthorized deployments, and compromised instances.
💡 A security guard who never sleeps, watching all your cameras and raising the alarm if anything looks wrong.

💡 Think of CloudTrail as the CCTV footage of your entire AWS account — you can always rewind and see exactly what happened.

💡 A riot shield that absorbs waves of attack traffic so your application stays online and responsive.

Inspect and filter all VPC traffic.
A managed firewall that inspects and filters network traffic flowing in and out of your VPC. Deep packet inspection, intrusion prevention, and web filtering — all without managing hardware.
💡 Airport security that checks everyone entering or leaving the country — scanning for prohibited items.

💡 A compliance auditor who watches your settings 24/7 and raises a red flag the moment something changes.

💡 A library of certificates that prove AWS meets regulatory standards — hand them to your auditor and move on.

The keymaster of your cloud kingdom.

Identity & Access Management

The most important security service in AWS. Master this first.

⭐⭐⭐ EXAM CRITICAL
ROOT
Root User ⚠️ Never use for daily work
Full account access. Cannot be restricted. The master key to everything.
USERS
IAM Users
Individual people. Permanent credentials. Each person gets their own login.
GROUPS
IAM Groups
Collections of users sharing the same permissions. Manage access at scale.
ROLES
IAM Roles
Temporary access. Best practice for applications and cross-account access.
POLICIES
IAM Policies
JSON rules that define exactly what is allowed or denied.
🎯CRITICAL PRINCIPLE

Principle of Least Privilege

Give every user and system ONLY the permissions they need — nothing more. A developer who deploys code should not have access to billing information. Start with zero permissions and add only what's required.

⚠️EXAM FAVOURITE

Never Use Root for Daily Work

The AWS root account is like the master key to your entire cloud infrastructure. Create IAM users for all daily tasks. Use root only to create your first IAM admin — then lock it away and enable MFA.

📱BEST PRACTICE

Enable MFA Everywhere

Multi-Factor Authentication adds a second verification step. Even if someone steals your password, they cannot log in without your phone. Enable it on root and all admin accounts — it's free and takes 2 minutes.

🔄PROFESSIONAL PRACTICE

Use Roles for Applications

When an EC2 server needs to access S3, don't give it a username and password. Assign an IAM Role — temporary, automatic, and far more secure than stored credentials that could be leaked.

example-iam-policy.json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "s3:GetObject",
      "s3:PutObject"
    ],
    "Resource": "arn:aws:s3:::my-company-bucket/*"
  }]
}

This policy allows reading and writing files to ONE specific S3 bucket — nothing else. This is least privilege in practice.

VPC Security Architecture

Your private network inside AWS — and how to protect it.

☁️ AWS Cloud
Your VPC (10.0.0.0/16)

🌐 Public Subnet

Accessible from the internet
🌍Internet Gateway
⚖️Application Load Balancer
🔀NAT Gateway

🔒 Private Subnet

No direct internet access
💻Application Server (EC2)
🗄️Database (RDS)
📦Cache (ElastiCache)
🔒 Security Groups — per resource
🧱 Network ACLs — per subnet
🛡️ WAF — in front of Load Balancer

🔒 Security Groups

The firewall built around each individual AWS resource. Acts like a personal bodyguard for each EC2 instance or RDS database. STATEFUL: if you allow inbound, the response is automatically allowed out.

DEFAULT: Deny everything unless you explicitly create an allow rule.

💡 A bouncer assigned to each specific person — they only let through what's on the approved list.

🧱 Network ACLs

The firewall at the subnet level — one layer outside Security Groups. Controls traffic entering or leaving an entire subnet, not just one resource.

STATELESS: you must explicitly allow both inbound AND outbound traffic separately.

💡 Border control at the entrance to an entire neighbourhood — checking everyone who enters and leaves.

🛡️ Defense in Depth

Real security uses MULTIPLE layers — WAF → NACL → Security Group. Even if one layer is misconfigured, the others still protect you. This is the core principle of enterprise cloud security.

Always use all layers together.

💡 A bank vault with a guard outside, a locked door, a keypad, and the vault itself — multiple barriers, maximum protection.

GDPR & Cloud Compliance on AWS

Why European cloud security is different — and why AWS is built for it. 🇪🇺

1

What GDPR Is

The General Data Protection Regulation (GDPR) is the EU law that governs how personal data is collected, stored, and processed. It applies to ANY company handling data of EU residents — regardless of where that company is located.

Fines: up to €20 million or 4% of global annual revenue, whichever is higher.

2

How It Affects AWS Architecture

Where you store data matters. Storing EU customer data in us-east-1 (Virginia) can violate GDPR. Best practice:

  • Use eu-west-3 (Paris) or eu-central-1 (Frankfurt) for EU data
  • AWS provides data residency controls to keep data in specific regions
  • Configure S3 bucket policies to enforce region restrictions
3

AWS Tools for GDPR

  • AWS Artifact — Download GDPR compliance certificates
  • AWS Config — Prove configs haven't changed since audit
  • CloudTrail — Full audit log of every action (required by GDPR)
  • AWS KMS — Encrypt personal data (GDPR requires this)
  • Amazon Macie — Auto-discover & protect personal data in S3
GDPR Principle AWS Implementation
Data Minimisation IAM policies limit who can access which data — principle of least privilege
Purpose Limitation S3 bucket policies restrict data access per use case and application
Storage Limitation S3 Lifecycle rules automatically delete data after retention period expires
Integrity & Confidentiality KMS encryption at rest + TLS in transit + CloudTrail audit logs
Accountability AWS Artifact compliance reports + CloudTrail audit trail = full accountability
Right to Erasure S3 Object Delete + RDS record removal procedures for "right to be forgotten"
Data Portability AWS Data Export tools + S3 APIs for structured data extraction

🇪🇺 European Cloud Security

Europe hosts some of the world's most regulated financial markets, with strict data protection requirements that every cloud professional must understand.

AWS professionals working with European clients must understand: GDPR, NIS2 Directive, and the EU's DORA regulation (Digital Operational Resilience Act).

Understanding these frameworks places you ahead of most cloud professionals. The intersection of cloud security knowledge and European regulatory awareness is extremely rare — and extremely valuable.
